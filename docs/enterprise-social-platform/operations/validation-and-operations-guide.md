# 验收与运营管理指南

这份文档用于本地、内测和准生产环境的功能验收，也作为 `/admin` 后台运营管理的使用手册。当前项目还处在推荐系统和内容运营能力快速建设期，后台里有些模块是“可直接生效”的业务操作，有些模块只是“登记/观测/追踪”，使用时要区分清楚。

## 1. 启动验收

1. 启动基础设施：MySQL、Redis、MinIO、Milvus，以及按需启动 Kafka、Elasticsearch。
2. 启动后端，确认 Liquibase 已自动执行到最新 changeset。
3. 启动深度排序服务，确认 `deep_rank.pt` 和 `candidate_generator.pt` 路径正确。
4. 启动前端，确认登录、发布、收藏、评论、关注、详情页、首页 Feed 正常。
5. 打开 `/admin`，使用具备 `ROLE_ADMIN` 的账号验证后台接口可访问。

常用页面入口：

| 场景 | 地址 | 说明 |
| :-- | :-- | :-- |
| 首页 Feed | `/home` 或 `/feed` | 首页推荐、关注、朋友动态、频道筛选 |
| 频道页 | `/channels/{channelCode}` | 单频道内容流 |
| 搜索页 | `/search` | 搜索内容、用户、频道 |
| 发布页 | `/publish` | 用户发布内容 |
| 详情页 | `/posts/{postId}` | 图片详情和相似推荐 |
| 个人页 | `/profile` 或 `/users/{userId}` | 用户资料和内容 |
| 管理后台 | `/admin` | 运营、内容、推荐、模型和治理 |

## 2. 后台权限与安全边界

后台页面只是入口保护，真正的权限必须以后端为准。当前后台接口统一走 `/api/admin/**` 和 `/api/admin/enterprise/**`，后端用 `ROLE_ADMIN` 鉴权。

安全底线：

1. 不要依赖前端隐藏入口来保护后台，后端鉴权不能删。
2. 生产环境不要把管理员账号、JWT secret、数据库密码写死在配置里。
3. 批量导入、批量回滚、内容下架、账号处罚属于高风险操作，操作前先确认范围。
4. 新模型、新推荐实验不要直接全量上线，先 shadow 或小流量观察。
5. 准商业阶段建议把后台拆成独立应用，例如 `admin.vibelo.com`，并在网关层加 IP 白名单、MFA、限流和审计。

## 3. 后台模块地图

| 模块 | 主要用途 | 是否直接影响线上 |
| :-- | :-- | :-- |
| 概览 | 看频道、话题、内容、导入批次数量 | 否 |
| 频道 | 新增/编辑频道，控制发布、推荐、瀑布流开关 | 是 |
| 话题 | 新增/编辑话题，别名、频道绑定、话题合并 | 是 |
| 内容 | 搜索内容，改审核状态、可见性、质量分和安全分 | 是 |
| 导入 | 建批次、录入内容、发布导入项、批量发布、回滚 | 是 |
| 重建 | 登记搜索、特征、向量、I2I 等重建任务 | 半自动，目前主要是登记 |
| 推荐 | 查看 Feed 请求日志和曝光日志 | 否 |
| 模型 | 登记训练集、模型版本、离线评估报告 | 半自动，目前主要是登记 |
| 治理 | 创建/处理内容治理案件，创建账号处罚动作 | 是 |
| 创作者 | 维护创作者档案、商业内容披露信息 | 是，但当前更多是标记和治理 |

## 4. 后台日常使用顺序

推荐按这个顺序做运营工作：

1. 先看“概览”，确认当前频道、话题、内容和导入批次数量是否异常。
2. 如果要扩内容结构，先配置“频道”，再配置“话题”，最后导入或发布内容。
3. 如果要处理内容风险，先在“内容”里定位帖子，再根据情况去“治理”里登记案件。
4. 如果导入了新内容，先在前台抽查详情页和首页，再视情况执行搜索、特征、I2I、embedding 等重建脚本。
5. 如果要观察推荐效果，先看“推荐”里的请求日志和曝光日志，再看深度排序服务日志。
6. 如果要登记模型版本，先创建训练集，再创建模型版本，再登记离线评估报告。

## 5. 概览模块

概览只用于快速判断系统状态，不用于编辑数据。

重点看：

1. 启用频道数是否符合预期。
2. 启用话题数是否异常减少。
3. 已通过内容数是否稳定增长。
4. 待审和拒绝内容是否突然飙升。
5. 导入批次数是否有异常堆积。

如果概览数字明显不对，先不要继续做批量操作，优先检查对应模块的筛选结果。

## 6. 频道管理

频道决定前台左侧导航、发布表单类型、频道页内容、推荐过滤和瀑布流展示方式。

常用字段：

| 字段 | 用法 |
| :-- | :-- |
| `code` | 稳定唯一编码，创建后不要随便改，例如 `campus`、`pet` |
| `name` | 前台展示名称 |
| `description` | 频道说明，会影响用户理解 |
| `icon` / `coverUrl` | 图标或封面 URL |
| `sortOrder` | 导航排序，小的靠前 |
| `status` / `enabled` | 是否启用 |
| `defaultPostType` | 默认发布类型，例如 `campus_post` |
| `publishEnabled` | 是否允许用户往该频道发布 |
| `recommendEnabled` | 是否允许进入推荐池 |
| `waterfallEnabled` | 前台是否按瀑布流展示 |
| `configJson` | 扩展配置，当前不熟悉时保持 `{}` |

新增频道建议流程：

1. 创建频道，先使用清晰短 code，不要用中文或空格。
2. 打开 `enabled`、`publishEnabled`、`recommendEnabled`。
3. 如果是图片内容，打开 `waterfallEnabled`。
4. 保存后调整排序。
5. 去 `/home` 和 `/channels/{code}` 验证入口和列表。
6. 发布一条测试内容，确认发布表单、详情页、频道页都正常。

慎用：

1. 不要随便改已有频道的 `code`，否则历史帖子、话题绑定和前端路由可能断。
2. 关闭 `recommendEnabled` 后，该频道内容可能不再进入首页推荐。
3. 关闭 `publishEnabled` 后，用户可能不能继续向该频道发内容。

## 7. 话题管理

话题用于内容归类、搜索、频道聚合、推荐召回和运营活动。话题治理比频道更细，容易影响推荐质量。

常用操作：

| 操作 | 说明 |
| :-- | :-- |
| 新建话题 | 创建一个可被内容绑定和搜索的主题 |
| 编辑话题 | 修改名称、描述、封面、状态、风险等级、热度 |
| 绑定频道 | 让话题归属到一个或多个频道 |
| 添加别名 | 让搜索、导入和语义归类识别更多写法 |
| 合并话题 | 把重复话题迁移到目标话题 |

新增话题建议流程：

1. 先确认是否已有相似话题，避免重复创建。
2. 新建话题时，`name` 用用户能看懂的名称，`slug` 用稳定英文或拼音短码。
3. `riskLevel` 默认用 `NORMAL`，敏感内容再调高。
4. 绑定至少一个频道，否则前台聚合和运营筛选会变弱。
5. 添加常见别名，例如中文名、英文名、缩写、错别字。
6. 保存后去搜索页和频道页抽查。

话题合并注意：

1. 合并会把来源话题的内容关系、别名、频道绑定迁移到目标话题。
2. 来源话题会被标记为 `MERGED`。
3. 合并前要确认目标话题是最终保留项。
4. 合并后建议重建搜索索引和语义/标签相关数据。

## 8. 内容管理

内容管理用于审核、下架、恢复、调质量分和安全分。

常见状态：

| 字段 | 推荐值 | 说明 |
| :-- | :-- | :-- |
| `auditStatus` | `APPROVED` | 审核通过 |
| `auditStatus` | `PENDING_REVIEW` | 待审 |
| `auditStatus` | `REJECTED` | 审核拒绝 |
| `visibility` | `PUBLIC` | 公开可见 |
| `visibility` | `PRIVATE` | 私有或回滚隐藏 |
| `visibility` | `HIDDEN` | 运营下架 |
| `qualityScore` | 0 到 1 | 内容质量分，影响排序 |
| `safetyScore` | 0 到 1 | 安全分，低分内容应谨慎推荐 |

处理问题内容：

1. 用关键词、频道、审核状态或可见性筛选帖子。
2. 先打开详情确认是不是误伤。
3. 轻微问题可以降低 `qualityScore` 或 `safetyScore`。
4. 明确违规内容设置为 `auditStatus=REJECTED` 且 `visibility=HIDDEN`。
5. 下架后刷新首页、搜索页、频道页，确认不再展示。
6. 如果是严重违规，到“治理”模块创建内容治理案件。

质量运营建议：

1. 优质图文可以提高 `qualityScore`，帮助规则排序兜底。
2. 低质、重复、无图、标题党内容不要只靠模型处理，应先在内容层清理。
3. 批量清理后建议重跑特征和 I2I，避免旧内容继续影响推荐。

## 9. 导入管理

导入模块用于运营批量造内容或导入冷启动内容。

导入流程：

1. 创建导入批次，填写批次名称、描述、来源类型。
2. 选中批次后添加导入项。
3. 每个导入项至少填写标题、正文、频道、话题和图片 URL。
4. 可以单条发布导入项，也可以发布整个批次。
5. 发布成功后，系统会调用正常发帖逻辑创建帖子。
6. 如果批次导入有问题，可以回滚批次。

回滚行为：

1. 回滚不会物理删除帖子。
2. 已生成帖子会被设置为 `visibility=PRIVATE`、`auditStatus=REJECTED`。
3. 回滚后仍建议去“内容”模块抽查。

导入后必做检查：

1. 抽查新帖详情页图片是否可打开。
2. 检查频道页是否能看到导入内容。
3. 检查首页是否逐步出现新内容。
4. 如果图片 URL 来自 MinIO，确认前端能通过 `/minio-img` 访问。
5. 视情况执行搜索索引、特征、I2I、embedding 相关重建。

## 10. 重建任务

当前后台“重建”模块主要用于登记任务、追踪状态、做运营协作记录。它本身不等于已经执行了 Python 脚本或容器任务。

任务类型参考：

| 类型 | 作用 | 当前建议 |
| :-- | :-- | :-- |
| `SEARCH_INDEX` | 重建搜索索引 | 导入或大批量改内容后执行 |
| `FEATURE` | 刷新用户/帖子特征 | 常驻维护任务已覆盖，必要时手动跑 |
| `I2I` | 刷新相似内容 | 常驻维护任务已覆盖，必要时手动跑 |
| `EMBEDDING` | 抽取图片向量 | 新增大量图片后手动跑 |
| `SEMANTIC` | 刷新语义分类和标签 | 调整频道/话题或导入大批内容后手动跑 |
| `THUMBNAIL` | 生成缩略图 | 图片资源缺缩略图时跑 |
| `ALL` | 全量流水线 | 谨慎使用，适合离线维护窗口 |

使用方式：

1. 在后台创建重建任务，写清楚 `taskType`、`scopeType`、`batchId`、`postId` 或参数。
2. 到服务器或本地执行对应脚本。
3. 执行中把任务状态改为 `RUNNING`。
4. 执行成功后填写总数、成功数、失败数，并改为 `SUCCESS`。
5. 执行失败时记录错误信息，并改为 `FAILED`。

当前常驻维护任务：

1. `feature`：刷新用户/帖子特征到 MySQL、Redis。
2. `i2i`：刷新协同过滤相似帖子。

当前建议手动执行：

1. `embedding`：图片向量抽取。
2. `taxonomy` / `semantic`：语义分类和 tag vocabulary。
3. `user-embeddings`：必要时刷新用户向量。

## 11. 推荐工作台

推荐模块用于排查“为什么首页展示这些内容”，不是直接编辑推荐结果。

重点看两类数据：

| 表 | 用途 |
| :-- | :-- |
| Feed 请求日志 | 每次 Feed 请求的 surface、page、size、实验桶、候选数、返回数、耗时 |
| 曝光日志 | 每个请求实际曝光了哪些 post、位置、召回来源、排序分、推荐原因 |

推荐效果排查流程：

1. 在前台刷新首页，记下大致时间和当前用户。
2. 到推荐模块按 `surface=home_feed` 查最新请求。
3. 点选请求后看曝光日志。
4. 检查返回数是否太少。
5. 检查召回来源是否单一，例如全是 hot 或 explore。
6. 检查推荐原因是否符合预期，例如 `From creators you follow`、`People with similar taste also liked`。
7. 如果耗时高，看后端日志和 deep-rank 服务日志。
8. 如果曝光重复，看最近曝光过滤、作者 cap、topic cap 和 I2I 数据。

常见问题：

1. 首页慢：优先看 deep-rank 是否超时、向量召回是否慢、MySQL 查询是否扫太多。
2. 推荐不准：看用户行为序列是否写入 Redis，特征任务是否成功，I2I 是否有数据。
3. 推荐重复：看去重、曝光过滤、作者上限和话题上限。
4. 新内容不出现：看内容是否 `PUBLIC + APPROVED`，频道是否 `recommendEnabled`，特征/I2I 是否刷新。

## 12. 模型管理

模型模块目前主要做训练集、模型版本、离线评估报告的登记和对账，不会自动训练模型。

推荐登记顺序：

1. 创建训练集，记录数据类型、路径、样本数、正负样本数。
2. 训练脚本跑完后，创建模型版本，记录模型名、版本号、模型类型、artifact 路径。
3. 创建离线评估报告，记录 AUC、NDCG、Recall、Precision 和报告路径。
4. 如果模型只准备观察，打开 `shadowEnabled`。
5. 如果准备上线，再配置 `onlineEnabled` 和 `trafficPercent`，但当前仍要以后端配置和服务实际加载为准。

重要提醒：

1. 后台登记模型不等于 deep-rank 服务已经加载该模型。
2. 线上生效仍取决于 `RANK_MODEL_PATH`、`CANDIDATE_MODEL_PATH` 和 deep-rank 服务启动状态。
3. 新模型上线前必须保留规则排序兜底。

## 13. 治理管理

治理模块用于内容案件和账号处罚。

内容治理流程：

1. 从前台举报或后台手动创建案件。
2. 填写 `postId`、原因、优先级、风险等级。
3. 审核内容详情和作者历史。
4. 如果内容没问题，处理为 `APPROVE` 或 `NO_ACTION`。
5. 如果内容违规，处理为 `REMOVE`，系统会同步下架对应帖子。
6. 处理后回到内容模块确认帖子状态。

账号处罚流程：

1. 输入用户 ID。
2. 选择动作，例如 `WARN`、`SUSPEND`、`BAN`。
3. 填写原因和可选过期时间。
4. 保存后系统会同步更新用户状态。
5. 对账号处罚要谨慎，建议保留完整原因。

## 14. 创作者与商业内容

创作者模块用于维护创作者等级、领域标签、质量分和变现状态。

创作者档案：

1. 输入用户 ID。
2. 填写领域标签，例如 `photography, campus, pet`。
3. 设置等级：`SEED`、`GROWTH`、`PRO`。
4. 设置质量分，范围 0 到 1。
5. 设置变现状态：`DISABLED`、`PENDING`、`ENABLED`。

商业内容：

1. 输入帖子 ID。
2. 填写品牌名、活动编码、披露类型。
3. 设置状态：`DRAFT`、`REVIEW`、`ACTIVE`。
4. 填写预算和落地页。
5. 当前系统更多是识别和治理商业内容，不建议默认强推广告。

## 15. 常见运营 Runbook

### 新建频道并上线

1. 在“频道”创建频道并启用。
2. 在“话题”创建 3 到 10 个初始话题并绑定频道。
3. 导入或发布一批冷启动内容。
4. 执行搜索索引、特征、I2I，必要时执行 embedding 和 semantic。
5. 检查 `/channels/{code}`、首页推荐、发布页和详情页。
6. 观察推荐请求日志，确认该频道开始进入曝光。

### 批量导入冷启动内容

1. 创建导入批次。
2. 添加导入项，保证每条内容有频道、话题和真实图片。
3. 先单条发布 1 到 3 条测试。
4. 前台抽查图片、详情页、相似推荐。
5. 批量发布剩余内容。
6. 跑搜索索引、特征、I2I，必要时跑 embedding。
7. 观察首页是否出现新内容。

### 处理违规内容

1. 在内容模块定位帖子。
2. 明确违规时设为 `REJECTED + HIDDEN`。
3. 在治理模块创建或处理案件。
4. 如涉及账号风险，创建账号处罚。
5. 前台确认 Feed、搜索、频道页不再展示。

### 排查首页推荐慢

1. 先确认后端和 deep-rank 服务都在运行。
2. 查看后端日志是否有 deep-rank timeout。
3. 在推荐模块看最新 Feed 请求耗时。
4. 看曝光日志是否返回数太少或来源单一。
5. 检查特征任务和 I2I 任务是否成功。
6. 必要时临时关闭或降低 deep-rank 超时，用规则排序兜底。

### 登记新模型版本

1. 跑训练脚本并保存模型和报告。
2. 在模型模块创建训练集记录。
3. 创建模型版本，写清 artifact 路径。
4. 创建离线评估报告。
5. 手动更新 deep-rank 服务模型路径并重启服务。
6. 先小流量或 shadow 观察，再扩大流量。

## 17. 上线后巡检

1. 执行 `backend/scripts/production_smoke_check.py` 检查健康、频道、话题、Feed。
2. 观察 API 延迟、错误率、Feed 空结果率、召回失败率。
3. 每日备份 MySQL、MinIO、Milvus、Elasticsearch。
4. 每次导入内容后重新检查搜索、特征、I2I、详情页图片和首页推荐。
5. 每天抽查后台“推荐”日志，确认请求量、曝光量和推荐来源没有异常。
6. 每周抽查“治理”和“内容”模块，清理违规、低质、重复内容。
